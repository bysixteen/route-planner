Sprint 002 · Full Sprint · 5 March 2026

Trip Viewer Data Concepts — Information Architecture

Format: Full (4 phases) Primary user: Daniel (Trip Planner) Scope: Sidebar information architecture Status: Complete
The Challenge

The trip viewer sidebar shows stops and drive times, but lacks the data concepts a trip planner needs: booking status, costs, fuel estimates, day grouping, outbound/return routes, stop purpose, and timeline awareness. Daniel can't answer "Is everything sorted?" from the sidebar alone.

Decision

Restructure the trip viewer sidebar into a three-layer information architecture: summary bar at top, day-grouped itinerary with booking status badges, and outbound/return section markers. All data derived from existing Supabase fields. Empty fields hidden, not dashed.

The Three Layers

Layer 1

Summary Bar (compact, always visible, max 60px)

Trip-level metrics answering "Is everything sorted?" at a glance:

  • Total distance & driving time
  • Total estimated cost (campsites + fuel)
  • Fuel estimate from vehicle data
  • Nights total
  • Days until departure (countdown)
  • Booking health: "X/Y confirmed"
  • Countries count
Layer 2

Day-Grouped Itinerary

Stops grouped by arrival date with day headers ("Day 3 — Monday 21 July"). Each stop card enriched with:

  • Booking status badge: Confirmed (green), Pending (amber), Not booked (red)
  • Stop type badge with context (existing)
  • Cost per stop (if entered)
  • Booking reference (if entered)
  • Falls back to flat list when dates are missing
Layer 3

Outbound / Return Markers

Section divider at the event stop (midpoint). Labels: "Outbound — X stops" / "Return — X stops". On the map: return route segments shown with dashed line style.

Booking Status Derivation

ConditionStatusBadge
booking_reference is non-nullConfirmedGreen
booking_reference is null AND type is campsite or transportPendingAmber
booking_reference is null AND type is city, attraction, rest, or eventNot applicableHidden

Rationale — Four Core Questions

Dr. Aris Thorne's framing was decisive: the sidebar must answer four questions without Daniel having to ask them. Every data concept was evaluated against these questions:

QuestionAnswered By
Is everything booked?Booking status badges + booking health count
What does this cost?Trip cost summary + fuel estimate
What do I need to do?"Needs action" count derived from data gaps
Where am I in the journey?Day grouping + outbound/return + countdown

What We Are NOT Doing

  • Booking management — display status only, no write operations from sidebar
  • Expense tracking — estimated costs, not actuals with receipts
  • Real-time fuel pricing — static consumption rate × distance × assumed price
  • Calendar sync or iCal export — day grouping is visual only
  • Task/todo management — "needs action" is a read-only count, not a checklist
  • Two-mode sidebar toggle — using visual hierarchy instead
  • Per-segment fuel breakdowns — total fuel estimate only
  • Mobile layout — deferred to Sprint 004

Success Criteria

  1. Daniel can state the booking status of any stop without clicking or expanding — booking badges visible on every bookable stop card
  2. Summary bar shows total cost, fuel estimate, and booking health (X/Y confirmed) without scrolling — all visible above the fold on 1080p
  3. Stops are grouped by day with day headers — Daniel can identify which stops belong to which travel day
  4. Outbound/return split is visually distinct — section marker and/or route line style change
  5. Empty data fields are invisible — no cost line if cost is null, no cost metric if all costs null
Next Action

Implement the creative brief — add the summary bar, day grouping, booking status badges, and outbound/return markers to the trip viewer sidebar. One schema addition: fuel_consumption_lper100km on the vehicles table.
