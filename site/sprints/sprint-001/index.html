Sprint 001 · Full Sprint · 5 March 2026

Map Interface UI — Rich Journey View

Format: Full (4 phases) Primary user: Daniel (desktop) Scope: Trip viewer only Status: Complete
The Challenge

The trip viewer map shows the route, but fails to communicate the journey — drive time and distance per leg are hidden behind popup clicks, there's no visual cue for the 5-hour constraint, and no discovery layer for the magical things the family will pass along the way.

Decision

Rebuild the trip viewer as a desktop two-panel layout — a 300px itinerary sidebar with segment times on the left, a full-height Mapbox outdoors map with colour-coded route segments and segment labels on the right. Add a gated POI discovery toggle and break stop markers.

What We're Building

01

Two-panel layout

300px fixed sidebar showing itinerary + segment times. Full-height map on the right. Replaces the tab system on desktop (≥md).

02

Segment time labels

Pill labels at each route segment midpoint: 4h 50m · 485km. Rendered as a Mapbox symbol layer. text-optional prevents overlap.

03

Colour-coded segments

Green <4h · Amber 4–5h · Red >5h. Data-driven from the existing segment duration. Current route should show all green/amber.

04

Mapbox outdoors style

Switch to mapbox://styles/mapbox/outdoors-v12. Mountains, lakes, forests, and elevation visible by default. No extra cost.

05

POI discovery toggle gated

Toggle in map controls. queryRenderedFeatures for landmarks, castles, viewpoints. Must be validated against the actual Cork route before shipping enabled.

06

Break stop markers

Small markers at leg midpoints, visible at zoom ≥ 7. Click shows "Suggested break — approx. Xh from [origin]." Subordinate to stop markers.

Rationale

Three findings drove this decision. First, the segment data already exists — duration and distance are returned by the Directions API in trip-map.tsx — making this a pure UI problem. Second, Nara's competitive research confirmed that co-visible sidebar + map is the industry-standard pattern for route planning on desktop; no serious route planning tool uses a tab system. Third, the Mapbox outdoors style and symbol layers support all required features without any new npm dependencies.

Dr. Aris Thorne's reframe was decisive: the core problem is not "the map lacks features" — it is "Daniel cannot confirm whether the trip is safe to drive without clicking every stop." Segment time visibility (features 01–03) directly solves this. The rest (POI toggle, break markers) are enhancements that can be gated on quality checks.

What We Are NOT Doing

  • Real-time traffic, live ETAs, or Waze-style navigation
  • Custom POI database — using Mapbox rendered features only (gated on quality)
  • Changes to editor-map.tsx — viewer and editor are separate concerns
  • Mobile-first or responsive sidebar design — Sprint 004
  • Navigation handoff (Google Maps / Apple Maps deep links) — Sprint 006
  • Offline mode — Sprint 006
  • Animated route drawing or journey playback
  • Multiple alternative routes

Success Criteria

  1. Daniel can state the drive time and distance for any leg without clicking a marker or opening a popup
  2. All legs on the current Cork route display as green or amber (zero red segments)
  3. At least two named POIs per leg visible when POI toggle is on — if not, toggle ships hidden
  4. The sidebar itinerary shows all 11 stops with segment times in a single vertical scroll at 1280px
Next Action

Run the implementation sprint. Build the two-panel layout, segment labels, colour coding, and Mapbox outdoors style switch in trip-map.tsx and app/trip/[id]/page.tsx. Gate the POI layer on data quality validation against the actual Cork route.
